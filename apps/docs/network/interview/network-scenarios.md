# 网络场景面试题

## 打开一个网页时发生了什么？

关键句：URL 解析 -> DNS 解析 -> 建连 -> TLS（可选）-> HTTP 请求响应 -> 渲染。

典型链路：

1. 浏览器解析 URL 和协议。
2. 命中缓存或发起 DNS 查询。
3. 通过 ARP/网关准备链路层转发。
4. 与目标建立 TCP 三次握手。
5. HTTPS 场景先完成 TLS 握手。
6. 发送 HTTP 请求并接收响应，浏览器解析渲染。

## 页面很慢，一直转圈，怎么排查？

关键句：按层定位，优先确定“慢在 DNS、连接、传输还是应用”。

排查顺序：

1. 客户端网络是否正常（是否能访问其他站点）。
2. DNS 是否解析成功。
3. TCP 是否成功握手。
4. 服务端响应码和耗时是否异常。
5. 前端静态资源、脚本执行是否阻塞渲染。

## 两台服务器如何确认网络连通？

关键句：先测网络，再测端口，最后测应用协议。

1. `ping` 验证基础可达性（ICMP）。
2. `traceroute` 定位中间路径问题。
3. `telnet`/`nc`/`curl` 验证目标端口与协议。
4. `ss -lntp` 检查服务监听状态。

## 服务启动了但客户端请求不到，常见原因有哪些？

关键句：高频在“地址端口、路由防火墙、代理配置、服务绑定地址”。

1. IP/端口填错。
2. 服务只监听 `127.0.0.1`，未监听外网网卡。
3. 防火墙或安全组未放行端口。
4. 客户端走了错误代理。
5. 网关/NAT/路由策略不通。

## ping 不通但 HTTP 能通，可能吗？

关键句：可能，ICMP 与 TCP 是不同协议栈路径。

1. 服务器或中间设备禁了 ICMP。
2. 80/443 端口仍然开放，所以 HTTP 正常。

## 服务端大量 TIME_WAIT 怎么定位？

关键句：通常是服务端频繁主动关闭连接。

高频原因：

1. 未使用或未稳定复用 HTTP 长连接。
2. Keep-Alive 超时太短。
3. 单连接请求上限过低，频繁换连接。

## 如何回答“排障思路”更像工程师？

关键句：给“现象 + 分层证据 + 结论 + 修复”。

模板示例：

1. 现象：请求超时，偶发 504。
2. 证据：握手成功但上游响应超时，网关日志一致。
3. 结论：应用层处理瓶颈而非基础网络不通。
4. 修复：扩容 + 超时参数优化 + 慢查询治理。
